# Melhores Práticas para Contribuições

## Regras de Commit
- Siga o padrão [Conventional Commits](https://www.conventionalcommits.org/pt-br/v1.0.0/):
    - Use prefixos como `feat:`, `fix:`, `docs:`, `chore:`, `refactor:`, etc.
    - Exemplo: `feat(api): adicionar endpoint de autenticação de usuário`
- Escreva mensagens de commit claras e concisas.
- Faça commits atômicos, contendo uma única alteração lógica por commit.
- Utilize branches para desenvolvimento e abra Pull Requests para revisão.
- Rebase ou squash os commits antes de fazer merge para manter o histórico limpo.

## Qualidade do Código e Revisões
- Siga o estilo de código e as regras de lint estabelecidos no projeto.
- Escreva e execute testes quando aplicável (unitários, integração).
- Revisões por pares são obrigatórias; os revisores devem verificar lógica, estilo e testes.
- Atenda a todo feedback antes de integrar o código.

## Documentação
- Mantenha a documentação clara, atualizada e objetiva.
- Atualize a documentação sempre que o código ou processos mudarem.
- Siga os padrões de localização se contribuir para traduções.
- Use formatação e terminologia consistentes.

## Colaboração
- Use o template de issues para abrir solicitações claras e focadas.
- Responda com cuidado e respeito a feedbacks em PRs e issues.
- Comunique-se de forma clara e respeitosa em todas as interações.

## CI/CD e Automação
- Garanta que todos os testes automáticos e verificações sejam aprovados antes do merge.
- Siga as diretrizes de deployment, se aplicáveis.
- Use os scripts e automações disponíveis para agilizar o workflow.
